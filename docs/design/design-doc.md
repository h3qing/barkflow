## Problem Statement

Voice transcription tools (Wispr Flow, TypeLess, OpenWhispr) are good at turning speech into text, but they stop there. The user still has to manually route the output: copy-paste into a todo app, open a browser, switch windows. The open-source landscape has two mature layers that exist independently — voice/transcription (OpenWhispr, Whispering, VoiceInk) and workflow automation (n8n, Activepieces, Huginn) — but nobody has built the bridge between them.

BarkFlow bridges that gap: speak a command, and it transcribes, polishes, routes, and executes — all in one flow. Named after Mando, Heqing's dog who listens faithfully and gets things done.

## What Makes This Cool

Three things no existing tool combines:

1. **Voice → Action, not just Voice → Text.** BarkFlow routes voice commands to workflows, not just to your cursor. "Add milk" goes to your grocery list. "Email Sarah about Friday" drafts the email.

2. **AI text polish, fully local.** Like Wispr Flow's clean output (filler removal, grammar, formatting) but running through a local LLM (Ollama) instead of cloud APIs. Your voice never leaves your machine.

3. **Unified capture layer.** Voice history + clipboard history in one searchable system. Everything you ever said or copied, retrievable and routable. No other tool unifies these.

## Constraints

- Must run fully local by default (no mandatory cloud dependency; architecture must not preclude optional cloud LLM in a future phase)
- Mac-first (macOS is the primary platform)
- English only for MVP (multi-language is a future consideration, not Phase 1)
- Fork of OpenWhispr (acknowledge upstream, maintain compatibility)
- Electron + React 19 + TypeScript + Tailwind v4 (inherited from OpenWhispr)
- Local LLM via Ollama for text polishing
- Plugin architecture for extensibility

## Premises

1. **Forking OpenWhispr is the right starting point** — it handles the voice layer (local Whisper/Parakeet STT, global hotkeys, cross-platform) well enough to focus on the workflow layer.
2. **Users are frustrated by manual routing** — voice-to-text tools produce text but leave the user to route it manually (copy-paste, switch windows, etc.).
3. **Mac-first desktop app is the right form factor** — voice automation lives in the OS layer (global hotkeys, menu bar), not in a browser.
4. **Local-first is a meaningful differentiator** — productivity nerds care about privacy and cost enough to choose local over polished cloud.
5. **Clipboard history and voice capture belong together** — they're the same fundamental action: capturing something for later use.

## Approaches Considered

### Approach A: Fork OpenWhispr + LLM Polish + Plugin System (CHOSEN)
Fork the full Electron app, add Ollama/local LLM as a post-processing step for AI text cleanup, add a plugin system for workflow routing, and integrate clipboard history. Same tech stack (Electron/React/TS/Tailwind).

- **Effort:** M (human: ~3 weeks / CC-assisted coding: ~3 hours, plus debugging/testing/integration time)
- **Risk:** Low
- **Pros:** Fastest to daily use, inherits proven STT + hotkeys + UI, fork keeps upstream compatibility
- **Cons:** Electron is heavier than native, plugin system bolted on existing code

### Approach B: Tauri App (Rust + React)
Build from scratch using Tauri (lighter than Electron, ~10x smaller binary). Use OpenWhispr's STT approach as reference. Better Mac integration, lower memory.

- **Effort:** L (human: ~8 weeks / CC-assisted coding: ~5 hours, plus debugging/testing/integration time)
- **Risk:** Medium
- **Pros:** Best architecture, native feel, Rust plugin system
- **Cons:** Much more upfront work, can't pull OpenWhispr improvements, Mac-only

### Approach C: Fork OpenWhispr + Cloud LLM Option
Same as A but also support cloud LLMs (OpenAI, Anthropic) for premium polish quality. Local Ollama as default, cloud as opt-in.

- **Effort:** M (human: ~3 weeks / CC-assisted coding: ~3 hours, plus debugging/testing/integration time)
- **Risk:** Low
- **Pros:** Gives users choice, same effort as A
- **Cons:** Cloud dependency complicates the "local-first" story

## Recommended Approach

**Approach A: Fork OpenWhispr + LLM Polish + Plugin System**

Rationale: Fastest path to a daily-use product. The plugin architecture is the key decision — get that right, and cloud LLM support (Approach C), visual flow builder, and other features can be added incrementally without rewriting. Local-first keeps the story clean.

### Core Pipeline

```
Voice Input (hotkey trigger)
    │
    ▼
OpenWhispr STT (local Whisper/Parakeet)
    │
    ▼
Raw Transcript
    │
    ▼
Local LLM Polish (Ollama — filler removal, grammar, formatting)
    │
    ▼
Polished Text
    │
    ├──▶ Default: Paste at cursor (like OpenWhispr today)
    ├──▶ Hotkey A: Route to Plugin A (e.g., Todoist)
    ├──▶ Hotkey B: Route to Plugin B (e.g., Notion)
    └──▶ Clipboard History (searchable archive)
```

### Core Interaction Model: Hotkey-Driven Intent

**The hotkey combination you hold determines the destination.** No LLM intent detection — the user is in explicit control.

- **Fn** (hold) → speak → paste polished text at cursor (default STT)
- **Fn** (double-click) → toggle hands-free mode (keep speaking without holding)
- **Fn + T** → speak → add to todo list
- **Fn + N** → speak → save as Markdown note to configurable directory
- **Fn + C** → speak → add to calendar
- Each combo is user-configurable in Settings → Hotkeys

**Floating indicator:** When speaking, a floating bar appears on screen showing recording status. Two customizable styles:
- **Classic** — standard waveform/pulse animation
- **Bark** — dog ear animation that perks up when you speak (on-brand, shareable)

### Phase 1 — MVP (the "can't live without it" version)

**OpenWhispr baseline:** OpenWhispr (Electron 39 + React 19 + Tailwind v4 + Vite) already provides: multi-engine STT (Whisper/Parakeet + cloud BYOK), LLM polishing via `ReasoningService` + `llamaServer.js`, macOS Globe/Fn key via native Swift binary, audio recording/playback via `AudioManager` + `audioStorage.js`, SQLite via better-sqlite3 + Kysely ORM, clipboard paste helpers (native Swift/C), system tray, onboarding flow, shadcn-ui components, notes system (TipTap), i18n (10 languages), and auto-updater. **OpenWhispr has ZERO tests** and has security gaps (`sandbox: false`, `webSecurity: false`, no CSP on control panel).

Items marked **[ADAPT]** extend existing OpenWhispr code. Items marked **[NEW]** are built from scratch. Items marked **[HARDEN]** fix existing issues.

1. **[ADAPT]** Fork OpenWhispr, rebrand to BarkFlow (update branding, README with fork attribution, app icons)
2. **[ADAPT]** Switch LLM polish from `llamaServer.js` (llama.cpp) to Ollama — rewrite `ReasoningService` to call Ollama HTTP API at `localhost:11434`. Keep fallback chain: Ollama → raw transcript.
3. **[ADAPT]** Extend `HotkeyManager` for routing — add destination routing to the existing Globe key listener. Current hotkeys: dictation/agent/meeting. New: configurable Fn+key combos → arbitrary destinations (paste, clipboard, markdown, todo).
4. **[NEW]** Clipboard monitoring via NSPasteboard polling — OpenWhispr has paste helpers but NOT clipboard capture/history. Build new `ClipboardMonitor` module with `ConcealedType` detection.
5. **[NEW]** StorageProvider abstraction — wrap existing Kysely/SQLite `database.js` behind `StorageProvider` interface. Add `entries` table, FTS5 index, `audit_log` table. Existing OpenWhispr tables (transcriptions, notes, dictionary, actions) remain as-is.
6. **[ADAPT]** Searchable history UI — OpenWhispr has `HistoryView.tsx` for transcriptions. Extend or replace with sidebar + detail pane layout, adding source filters (voice/clipboard), date range, FTS search.
7. **[ADAPT]** Audio playback in history — OpenWhispr's `AudioStorageManager` already stores audio. Wire it into the new history UI with inline player component.
8. **[ADAPT]** Storage/cache management UI — OpenWhispr has settings infrastructure. Add new settings sections for disk usage, retention, cache management per the settings screen spec.
9. **[ADAPT]** Floating indicator reskin — OpenWhispr has a 96×96 always-on-top dictation panel. Reskin to BarkFlow indicator with Classic + Bark (dog ear) styles per the state machine spec.
10. **[NEW]** Voice-to-Markdown routing (Fn+N → polished .md file to configurable directory)
11. **[NEW]** Projects ("wandering mind" capture) — hotkey combo (e.g., Fn+P) from any interface routes voice input to a named Project bucket instead of the current cursor. Projects are persistent collections of captured thoughts — review later, dispatch to integrations, or send to agents via MCP. Each project has a name, entries list, and optional integration target. Projects live in the StorageProvider alongside regular entries (tagged with `project_id`). UI: project picker in sidebar, project detail view, "dispatch" button per entry.
12. **[ADAPT]** Meeting recording — OpenWhispr already has `AudioTapManager` (macOS CoreAudio tap for system audio capture) and meeting-related features. Adapt for BarkFlow: record system audio locally (like Granola), transcribe via STT pipeline, save to history as `source: 'meeting'`. Add **transcript-only mode** option: capture and transcribe but don't persist audio file — saves storage. Settings toggle: "Save meeting audio" (on/off, default: on).
13. **[NEW]** File upload pipeline — accept local audio files (phone recordings, voice memos, .mp3/.m4a/.wav/.webm) and run through the same STT → Polish → Route pipeline. Drop zone in history UI + menu bar "Import audio..." option. Transcoded to compatible format via ffmpeg (already bundled in OpenWhispr). Entry tagged with `source: 'import'` and original filename in metadata.
14. **[HARDEN]** Security: fix `sandbox: false` and `webSecurity: false` on control panel (proxy cloud API calls through main process), add CSP, full preload bridge audit (~100+ methods → explicit allowlist), encrypt audio files, set file permissions.
15. **[NEW]** Testing infrastructure — OpenWhispr has zero tests. Set up Vitest, write tests for StorageProvider, pipeline, routing, clipboard monitor, projects. Target: 80%+ coverage on new BarkFlow code.

### Failure Modes

| Step | Failure | User Sees | Fallback |
|------|---------|-----------|----------|
| STT (Whisper/Parakeet) | Model fails to load | Error toast: "STT model not loaded" | Prompt to re-select model in settings |
| STT | Timeout (>10s silence) | Recording indicator stops | Discard, user re-triggers hotkey |
| STT | Garbled/empty result | Nothing pasted | Show raw audio duration in history, flag as "transcription failed" |
| LLM Polish (Ollama) | Ollama not installed | First-run setup prompt: "Install Ollama for AI text polish (optional)" | Skip polish, use raw transcript directly |
| LLM Polish | Ollama not running | Toast: "Starting Ollama..." (auto-start attempt) | If auto-start fails, skip polish, paste raw transcript |
| LLM Polish | Timeout (>2s) | Paste raw transcript immediately | Log timeout, suggest smaller model in settings |
| LLM Polish | Malformed response | Use raw transcript | Log error for debugging |
| Clipboard capture | macOS permission denied | System prompt for Accessibility access | History works for voice only until permission granted |
| Plugin (Phase 2) | Plugin unreachable/fails | Toast: "Todoist plugin failed — text saved to history" | Text always saved to history as fallback |
| Meeting STT | Streaming STT disconnects mid-meeting | Toast: "Transcription interrupted — reconnecting..." | Save partial transcript immediately, attempt auto-reconnect, resume. Gap in transcript shown as "[gap — reconnecting]" marker. If reconnect fails after 3 attempts, save what we have + toast: "Meeting transcript saved (partial)." |
| File Import | ffmpeg fails (corrupt/unsupported file) | Entry created with "failed" status in history | Detail pane shows error message + original filename. "Retry" button re-attempts transcode. "Remove" button deletes the failed entry. Supported formats listed in import dialog. |
| File Import | STT fails after successful transcode | Entry shows "transcription failed" status | Audio preserved (if not transcript-only mode). "Retry transcription" button re-runs STT only. |

### Ollama Lifecycle Management

BarkFlow manages Ollama as an optional dependency:
1. **Detection:** On startup, check `which ollama` and `curl localhost:11434/api/tags`
2. **Not installed:** Show first-run card: "Install Ollama for AI text polish (recommended). Without it, BarkFlow pastes raw transcripts." Link to ollama.com. BarkFlow works without it.
3. **Installed but not running:** Auto-start via `ollama serve` in background. Show "Starting Ollama..." toast.
4. **Running but model not pulled:** Auto-pull the configured model on first use. Show progress.
5. **Settings:** User can select model (default: `llama3.2:3b` for speed, recommend `llama3.1:8b` for quality), disable polish entirely, or configure a custom Ollama endpoint.

### Data Model

#### Storage Architecture: Repository Pattern

All data access goes through a `StorageProvider` interface. The app never talks to a database directly — it talks to the provider. This enables local-first (free) and cloud-synced (paid tier) without refactoring.

```typescript
interface StorageProvider {
  // Core CRUD
  saveEntry(entry: Omit<Entry, 'id'>): Promise<Entry>;
  getEntry(id: string): Promise<Entry | null>;
  deleteEntry(id: string): Promise<void>;

  // Update
  updateEntry(id: string, updates: Partial<Omit<Entry, 'id'>>): Promise<Entry>;

  // Search & query
  search(query: string, filters?: SearchFilters): Promise<Entry[]>;
  getRecent(limit: number, offset?: number): Promise<Entry[]>;

  // Migration
  exportAll(): AsyncIterable<Entry>;    // stream all entries out
  importAll(entries: AsyncIterable<Entry>): Promise<ImportResult>;

  // Lifecycle
  initialize(): Promise<void>;
  close(): Promise<void>;
}

interface Entry {
  id: string;
  createdAt: string;           // ISO 8601
  source: 'voice' | 'clipboard' | 'meeting' | 'import';
  rawText: string | null;
  polished: string | null;
  routedTo: string | null;
  hotkeyUsed: string | null;
  durationMs: number | null;
  projectId: string | null;    // null = no project, string = belongs to a Project
  metadata: Record<string, unknown>;
}

interface Project {
  id: string;
  name: string;
  createdAt: string;           // ISO 8601
  integrationTarget: string | null;  // Phase 2: MCP plugin ID to dispatch to
  metadata: Record<string, unknown>;
}

interface SearchFilters {
  source?: 'voice' | 'clipboard';
  dateFrom?: string;
  dateTo?: string;
  limit?: number;
  offset?: number;
}
```

#### Phase 1 Provider: SQLite (local)

Default storage via better-sqlite3 in Electron main process. Single database at `~/.barkflow/history.db`.

```sql
CREATE TABLE entries (
  id          TEXT PRIMARY KEY,  -- UUID for portability across providers
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  source      TEXT NOT NULL CHECK (source IN ('voice', 'clipboard', 'meeting', 'import')),
  raw_text    TEXT,
  polished    TEXT,
  routed_to   TEXT,
  hotkey_used TEXT,
  duration_ms INTEGER,
  audio_path  TEXT,             -- path to audio file (null if transcript-only mode)
  project_id  TEXT REFERENCES projects(id) ON DELETE SET NULL,
  metadata    TEXT              -- JSON blob
);

CREATE TABLE projects (
  id                  TEXT PRIMARY KEY,
  name                TEXT NOT NULL,
  created_at          TEXT NOT NULL DEFAULT (datetime('now')),
  integration_target  TEXT,    -- Phase 2: MCP plugin ID
  metadata            TEXT     -- JSON blob
);

CREATE INDEX idx_entries_created ON entries(created_at);
CREATE INDEX idx_entries_source ON entries(source);
CREATE INDEX idx_entries_project ON entries(project_id);
CREATE VIRTUAL TABLE entries_fts USING fts5(raw_text, polished, content=entries);
```

#### Future Providers (paid tier)

| Provider | Model | User Setup |
|----------|-------|------------|
| **BarkFlow Cloud** | BarkFlow hosts the database. User pays monthly. | Sign up, get token, paste into settings. |
| **BYOD Supabase** | User connects their own Supabase project. | Paste Supabase URL + anon key into settings. |
| **BYOD Neon/PlanetScale** | User connects their own Postgres-compatible DB. | Paste connection string into settings. |

Each provider implements the same `StorageProvider` interface. The app doesn't care which one is active.

#### Data Migration

Users can move between storage backends at any time:

```
SQLite (local) ←→ BarkFlow Cloud ←→ BYOD Supabase
```

Migration flow:
1. User selects new provider in settings
2. BarkFlow calls `currentProvider.exportAll()` → streams entries
3. BarkFlow calls `newProvider.importAll(stream)` → writes to new backend
4. Progress bar shows migration status (entry count + ETA)
5. On success: switch active provider, keep old data as backup
6. On failure: roll back, keep current provider, show error

The `exportAll()` / `importAll()` methods on the interface make this a first-class operation, not a hack.

### macOS Clipboard Access

macOS clipboard monitoring requires:
- **NSPasteboard polling** (every 0.5s check `NSPasteboard.general.changeCount`) — this is how Maccy works. No special entitlement needed.
- **Accessibility permission** is NOT required for clipboard reading — only for advanced paste operations (simulating Cmd+V into other apps).
- **Sequoia (macOS 15+):** No new restrictions on NSPasteboard polling. The Transparency, Consent, and Control (TCC) framework does not gate clipboard read access from foreground/background apps with user-installed status.
- **Privacy UX:** Show a brief explanation on first clipboard capture: "BarkFlow is monitoring your clipboard to build a searchable history. All data stays local."

### Floating Indicator — State Machine & Animation

The floating indicator is the only UI visible during the voice pipeline. It must communicate five distinct states without text — animation alone carries the meaning.

#### States

| State | Trigger | Dog Ear (Bark style) | Classic style | Duration |
|-------|---------|---------------------|---------------|----------|
| **Idle** | App running, not recording | Hidden (off-screen) | Hidden | — |
| **Ready** | Hotkey pressed, mic acquiring | Ears relaxed, fade in (150ms) | Dot appears, soft pulse | <200ms |
| **Recording** | Audio streaming from mic | Ears perked up, gentle bob on voice amplitude | Waveform animates with mic input | User-controlled (hold hotkey) |
| **Processing** | Hotkey released, STT + polish running | Ears tilt to side, subtle thinking pulse (0.8s cycle) | Waveform freezes → morphs to spinning dots | 1-3s typical |
| **Done** | Text pasted / routed | Ears perk briefly (happy), then fade out (300ms) | Dot flashes green, fade out (300ms) | 500ms visible |
| **Error** | Any pipeline failure | Ears droop, shake once, fade out | Dot flashes red, fade out | 1s visible + toast |

#### Transitions

```
[Idle] ──hotkey down──▶ [Ready] ──mic acquired──▶ [Recording]
                                                       │
                                                  hotkey up
                                                       │
                                                       ▼
                                                  [Processing]
                                                    │      │
                                              success    failure
                                                │          │
                                                ▼          ▼
                                             [Done]    [Error]
                                                │          │
                                            fade out   fade out
                                                │          │
                                                ▼          ▼
                                              [Idle]     [Idle]
```

#### Animation Specs

- **Position:** Bottom-center of active screen, 48px from bottom edge. Follows active display on multi-monitor setups.
- **Size:** 48×48px default (configurable: S 32px, M 48px, L 64px).
- **Dog ear asset:** Vector (SVG or Lottie) for crisp rendering at all sizes. Two ear shapes: left ear, right ear. Animate independently for personality.
- **Recording amplitude:** Map mic RMS level (0.0–1.0) to ear bob amplitude (0–4px vertical displacement). Smooth with 60ms lerp to avoid jitter.
- **Processing pulse:** CSS `opacity` oscillation 0.6–1.0 on a 0.8s ease-in-out loop. No layout shift.
- **`prefers-reduced-motion`:** Replace all animation with instant state changes (opacity snap, no bob/pulse/shake). Ears still change pose but don't animate between poses.
- **Always-on-top:** Indicator window is a frameless, transparent, always-on-top BrowserWindow. Click-through (`setIgnoreMouseEvents(true)`) so it never steals focus.

### Security

Addresses the 5 CRITICAL and 7 HIGH findings from the security review.

#### Data Encryption at Rest

Voice recordings are biometric data. Clipboard captures may contain passwords.

- **SQLite database:** Not encrypted at the application level. Relies on macOS FileVault for at-rest disk encryption. FTS5 full-text search indexes `raw_text` and `polished` fields directly — application-level field encryption would break search. Trade-off: any process running as the current user can read `history.db`. FileVault protects against physical theft and offline access.
- **Audio files:** Encrypt `.webm` files at rest using a key derived from Electron's `safeStorage` API (backed by macOS Keychain). Decrypt into memory for playback — never write decrypted audio to disk. Voice recordings are biometric data and warrant stronger protection than text.
- **Encryption key lifecycle:** Key for audio encryption is derived on first launch via `safeStorage`. No user-managed keys. If Keychain access is lost, audio files are unrecoverable (text history remains accessible).
- **Future consideration:** SQLCipher (whole-database encryption) can be added later if the threat model requires protecting text from other local processes. It's a drop-in replacement for better-sqlite3 and transparent to FTS5.

#### Clipboard Password Detection

Password managers (1Password, Bitwarden, macOS Passwords) set `org.nspasteboard.ConcealedType` on clipboard entries containing secrets.

- **Detection:** Check for `ConcealedType` flag on every clipboard poll. If present, skip capture entirely.
- **Transient entries:** Also check `org.nspasteboard.AutoGeneratedType` — skip auto-generated passwords.
- **User override:** No override. Concealed entries are never captured. This is a hard rule, not a preference.
- **Logging:** Log that a concealed entry was skipped (timestamp only, no content) for debugging.

#### Electron Hardening

Must verify after forking OpenWhispr. If any of these are wrong, fix before writing BarkFlow code.

```javascript
// Required webPreferences for ALL BrowserWindows
{
  nodeIntegration: false,
  contextIsolation: true,
  sandbox: false,              // Cannot enable — native modules (better-sqlite3,
                               // Swift binaries, ffmpeg) require unsandboxed main process.
                               // Security relies on IPC hardening instead.
  webSecurity: true,           // Enable — proxy all cross-origin requests through main process
  allowRunningInsecureContent: false,
  enableRemoteModule: false,   // deprecated but ensure it's off
}
```

**Sandbox reality:** OpenWhispr's renderer uses `sandbox: false` because native modules need direct Node.js access. Enabling sandbox would require restructuring all native access (database, hotkeys, clipboard, audio, ffmpeg) to go through IPC — a major rewrite. Instead, security is enforced at the IPC boundary: strict CSP, reduced preload bridge (full audit), typed channel allowlist, and main-process-only native access.

- **Content Security Policy:** Set strict CSP in the main window HTML:
  ```
  default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' http://localhost:11434
  ```
  The `localhost:11434` exception is for Ollama only. No other external connections in Phase 1.
- **IPC surface:** All main↔renderer communication through typed `contextBridge` / `ipcRenderer.invoke`. No `ipcRenderer.send` (fire-and-forget is harder to audit). Define an explicit allowlist of IPC channels.
- **No `shell.openExternal` with user input.** URLs opened in browser must come from a hardcoded allowlist (e.g., ollama.com, github.com/barkflow).

#### Ollama Security

- **Port verification:** Before sending any data to `localhost:11434`, verify the listening process is actually Ollama by checking process name via `lsof -i :11434` or equivalent. A malicious process on the same port could intercept all transcripts.
- **Request proxying:** BarkFlow's main process proxies all Ollama requests (renderer never talks to Ollama directly). Add a `X-BarkFlow-Internal: <nonce>` header — though this is defense-in-depth, not a security boundary.
- **SSRF mitigation:** Ollama's default `Access-Control-Allow-Origin: *` means any webpage could hit the local API. This is an Ollama issue, not BarkFlow's to fix, but document it as a known risk and recommend users set `OLLAMA_ORIGINS` restrictively.

#### File System Permissions

```bash
~/.barkflow/                 # 0700 — only user can access
  history.db                 # 0600 — encrypted SQLite
  audio/                     # 0700
    {id}.webm.enc            # 0600 — encrypted audio
  plugins/                   # 0700 — Phase 2
  .metadata_never_index      # Exclude from Spotlight indexing
```

- Set permissions on first launch. Verify on every startup — if permissions have been widened, warn the user and offer to fix.

#### Audit Log

Add an `audit_log` table from day one. Not user-facing in Phase 1, but invaluable for debugging and future compliance.

```sql
CREATE TABLE audit_log (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp  TEXT NOT NULL DEFAULT (datetime('now')),
  action     TEXT NOT NULL,  -- 'entry_created', 'entry_deleted', 'entry_exported', 'provider_switched', 'plugin_executed'
  entity_id  TEXT,           -- entry ID or plugin ID
  detail     TEXT            -- JSON blob with context (no sensitive content)

```

#### MCP Plugin Security (Phase 2 — design now)

- **Permission model:** Each MCP server plugin declares required permissions at install time: network domains it will contact, data types it will receive, filesystem paths it needs. User approves explicitly.
- **Data minimization:** Plugins receive only the text routed to them, not full history or other entries.
- **Network allowlist:** Plugins can only contact declared domains. BarkFlow proxies outbound requests and blocks undeclared destinations.
- **Sandbox:** Run MCP servers in a separate process with restricted filesystem access (only `~/.barkflow/plugins/{plugin-id}/`).
- **Revocation:** User can revoke plugin permissions at any time in Settings → Plugins → [plugin] → Permissions.

### Plugin API Shape (Phase 2)

```typescript
interface BarkFlowPlugin {
  id: string;                    // e.g., "todoist"
  name: string;                  // e.g., "Todoist"
  description: string;
  version: string;

  // Called when user routes text to this plugin
  execute(input: PluginInput): Promise<PluginResult>;

  // Optional: settings UI for configuration (API keys, defaults)
  getSettingsSchema?(): JSONSchema;
}

interface PluginInput {
  text: string;                  // polished text (or raw if polish disabled)
  rawText: string;               // always the raw STT transcript
  metadata: {
    timestamp: string;           // ISO 8601
    source: 'voice' | 'clipboard';
    hotkeyUsed: string;         // e.g., "Cmd+Shift+T"
    durationMs?: number;        // voice recording duration
  };
}

interface PluginResult {
  success: boolean;
  message?: string;              // shown as toast to user
  url?: string;                  // optional: link to created item (e.g., Todoist task URL)
}
```

Plugins are Node.js modules loaded from `~/.barkflow/plugins/`. Each plugin is a directory with a `package.json` and an entry point exporting a `BarkFlowPlugin`.
